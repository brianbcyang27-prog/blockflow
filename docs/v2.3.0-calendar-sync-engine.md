## 1. Current Architecture

### 1.1 Event Storage

**localStorage** is the primary storage, with Firebase as a secondary backup:

```
localStorage key: 'blockflow_calendar_events'
Schema: Array<Event>
```

**Current Event Object:**
```javascript
{
  id: 'evt_{timestamp}_{random}',      // Generated locally
  title: string,
  date: 'YYYY-MM-DD',
  time: 'HH:MM',                       // 24h format, optional
  endTime: 'HH:MM',                    // optional
  description: string,
  importance: 'high' | 'medium' | 'low',
  block: 'focus' | 'personal' | 'recovery',
  aiAnalyzed: boolean,
  createdAt: number,                   // Unix timestamp
  // Google Calendar import fields (optional):
  googleEventId: string,
  googleLink: string
}
```

**Storage API (js/storage.js):**
- `Storage.getCalendarEvents()` → Returns array from localStorage
- `Storage.saveCalendarEvents(events)` → Writes to localStorage + Firebase
- `Storage.addCalendarEvent(event)` → Creates with generated ID, saves, returns new event
- `Storage.updateCalendarEvent(eventId, updates)` → Merges updates, saves
- `Storage.deleteCalendarEvent(eventId)` → Filters out, saves

### 1.2 Firebase Schema

**Firestore Collections:**
```
users/{uid}/events/{eventId}      → Full event object
users/{uid}/data/history          → { records: [...] }
users/{uid}/data/settings         → settings object
users/{uid}/data/aiHistory        → { messages: [...] }
users/{uid}/data/aiMemory         → { memoryPoints: [...] }
users/{uid}/metadata/migration    → { completed, version, completedAt }
```

**Firestore Rules:**
```
match /users/{userId}/{document=**} {
  allow read, write: if request.auth != null && request.auth.uid == userId;
}
```

### 1.3 Google Calendar Integration

**OAuth Configuration:**
- Client ID: `255025772243-nmsk5afr14rgitlv3uj64goo0r8fuc90.apps.googleusercontent.com`
- Scope: `https://www.googleapis.com/auth/calendar.readonly` (READ-ONLY)
- Token storage: In-memory only (not persisted)

**API Endpoint:**
```
GET https://www.googleapis.com/calendar/v3/calendars/primary/events
Parameters: maxResults=100, singleEvents=true, orderBy=startTime, timeMin, timeMax
```

**Import Flow:**
1. User clicks "Import from Google Calendar"
2. OAuth popup → token received
3. Fetch events from Google (7 days past to 30 days future)
4. Transform Google event → BlockFlow event
5. Deduplicate by `googleEventId`
6. Save to localStorage via `Storage.saveCalendarEvents()`
7. Refresh calendar UI

**Data Transformation (Google → BlockFlow):**
| Google Field | BlockFlow Field |
|---|---|
| `ge.id` | `googleEventId` |
| `ge.summary` | `title` |
| `ge.start.dateTime` | `date` + `time` |
| `ge.start.date` | `date` (all-day) |
| `ge.end.dateTime` | `endTime` |
| `ge.description` | `description` |
| `ge.htmlLink` | `googleLink` |

**Data Access Points:**

## 2. Problems with Current Architecture

### 2.1 No Bidirectional Sync
- Google Calendar import is **read-only** (scope: `calendar.readonly`)
- Events created in BlockFlow never sync back to Google
- Changes made in Google after import are not reflected

### 2.2 No Incremental Sync
- No `syncToken` usage — full re-download every import
- Always fetches 7 days past + 30 days future
- Wastes API quota, slow for large calendars

### 2.3 No Token Persistence
- OAuth tokens stored in-memory only
- Lost on page reload — user must re-authenticate
- No refresh token handling

### 2.4 No Conflict Resolution
- `syncFromFirestore()` blindly overwrites localStorage
- Simultaneous edits on multiple devices cause data loss
- No `updatedAt` timestamps to determine which version is newer

### 2.5 localStorage as Primary Storage
- Not shared across devices
- Lost if user clears browser data
- Firebase is backup, not source of truth

### 2.6 AI Reads localStorage Only
- Nova can't see events from other devices
- No unified data layer for AI access

### 2.7 No Offline Queue
- If Firebase write fails, data is in localStorage but not cloud
- No mechanism to retry failed operations

### 2.8 No Multi-Calendar Support
- Only imports from `primary` calendar
- Can't sync multiple Google calendars
- No support for other providers (Apple, Outlook, Notion)

## 3. Proposed Architecture

### 3.1 Core Principle
**BlockFlow is the source of truth.** Google Calendar becomes one of many synchronized providers.

### 3.2 New File Structure
```
js/
  sync-engine.js          ← NEW: Core sync orchestrator
  providers/
    google-calendar.js    ← REWRITE: Two-way sync with incremental tokens
    apple-calendar.js     ← FUTURE: CalDAV sync
    outlook-calendar.js   ← FUTURE: Microsoft Graph API
  calendar.js             ← MODIFY: Read from unified event store
  storage.js              ← MODIFY: Add unified event methods
  ai-assistant.js         ← MODIFY: Read from unified event store
```

### 3.3 Unified Event Model
```javascript
{
  // Identity
  id: string,                    // Local ID: 'evt_{timestamp}_{random}'
  provider: string,              // 'local' | 'google' | 'apple' | 'outlook'
  providerEventId: string,       // Google event ID, Apple UID, etc.
  calendarId: string,            // Google calendar ID, etc.
  
  // Core fields
  title: string,
  description: string,
  location: string,
  start: string,                 // ISO 8601 datetime
  end: string,                   // ISO 8601 datetime
  allDay: boolean,
  
  // BlockFlow-specific
  color: string,
  importance: 'high' | 'medium' | 'low',
  block: 'focus' | 'personal' | 'recovery',
  
  // Sync metadata
  syncState: 'synced' | 'pending' | 'conflict' | 'error',
  lastSynced: number,            // Unix timestamp
  syncToken: string,             // Provider-specific sync token
  version: number,               // Optimistic concurrency control
  
  // Timestamps
  createdAt: number,
  updatedAt: number,
  
  // Soft delete
  deleted: boolean,
  deletedAt: number | null
}
```

### 3.4 SyncEngine Module (js/sync-engine.js)
```javascript
const SyncEngine = {
  // State
  providers: Map(),              // Connected providers
  eventStore: Map(),             // Local event cache
  syncQueue: [],                 // Offline operation queue
  
  // Core methods
  async connectProvider(provider, credentials),
  async disconnectProvider(providerId),
  
  // Sync operations
  async syncDown(providerId),    // Pull changes from provider
  async syncUp(providerId),      // Push changes to provider
  async syncAll(),               // Sync all connected providers
  
  // Event operations (unified API)
  async createEvent(event),
  async updateEvent(eventId, updates),
  async deleteEvent(eventId),
  async getEvents(filter),
  
  // Conflict resolution
  async resolveConflict(eventId, resolution),
  
  // Offline queue
  async queueOperation(operation),
  async replayQueue(),
  
  // Sync tokens
  async saveSyncToken(providerId, token),
  async getSyncToken(providerId)
}
```

## 4. Migration Strategy

### 4.1 Automatic Migration
- On first load after v2.3.0, detect old event format
- Transform existing events to unified model
- Set `provider: 'local'` for all existing events
- Preserve all existing fields
- Store migration version in Firebase metadata

### 4.2 Data Mapping
| Old Field | New Field |
|---|---|
| `id` | `id` (keep) |
| `title` | `title` (keep) |
| `date` + `time` | `start` (combine to ISO 8601) |
| `endTime` | `end` (combine to ISO 8601) |
| `description` | `description` (keep) |
| `importance` | `importance` (keep) |
| `block` | `block` (keep) |
| `createdAt` | `createdAt` (keep) |
| `googleEventId` | `providerEventId` |
| `googleLink` | (metadata, not stored) |
| — | `provider: 'local'` or `'google'` |
| — | `calendarId: 'primary'` (for Google) |
| — | `syncState: 'synced'` |
| — | `lastSynced: createdAt` |
| — | `version: 1` |
| — | `updatedAt: createdAt` |
| — | `deleted: false` |

### 4.3 Rollback Strategy
1. Keep old `Storage.getCalendarEvents()` working during transition
2. SyncEngine falls back to old methods if new methods fail
3. Feature flag: `blockflow_sync_engine_v2` in localStorage
4. If critical failure, disable SyncEngine and revert to localStorage-only mode

### Phase 5: Incremental Sync
- Store `nextSyncToken` per calendar
- Implement paginated sync with token persistence
- Handle 410 Gone with full resync

### Phase 6: Offline Queue
- Queue operations when offline
- Replay on reconnect with exponential backoff
- Store queue in localStorage (survives restarts)

### Phase 7: Conflict Resolution
- Compare `updatedAt` timestamps
- Last-write-wins for simple conflicts
- Prompt user for complex conflicts (same event edited differently)

### Phase 8: Sync Status UI
- Show sync status indicator (synced/syncing/error/offline/conflict)

## 6. Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Google API quota exhaustion | High | Implement backoff, cache aggressively |
| Sync token invalidation (410) | Medium | Full resync with merge strategy |
| Data loss during migration | High | Backup before migration, rollback flag |
| Conflict resolution complexity | Medium | Start with last-write-wins |
| Offline queue corruption | Low | Validate on replay, idempotency keys |
| Performance with large calendars | Medium | Pagination, background sync, caching |
